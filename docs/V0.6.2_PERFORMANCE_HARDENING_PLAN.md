# ClaraCore Desktop v0.6.2 Performance Hardening Plan

> Status: implemented and locally validated
> Target version: `0.6.2`
> Public release boundary: none; current public release remains `0.5.8`

## 1. Evidence From The Installed v0.6.1 App

The installed Lite app passed the maintained everyday workload:

- installed startup to ready: about 1.03 seconds;
- runtime overview p95: 22.3 ms on a consistent 43 MB database copy;
- shared HTTP 1/4/8-Agent p95: 8.9/18.1/39.3 ms with zero failures;
- stdio SQLite 1/4/8-Agent p95: 3.8/7.0/17.6 ms with zero failures and
  zero lock failures;
- five-minute installed endurance: 118 successful calls and 12.4 MiB RSS
  growth, below the 64 MiB short-run threshold;
- Memory Controller: 45 observed events and about 16 KiB before the installed
  test, already bounded by age, row, and byte retention.

Two remaining boundaries failed or showed future scaling risk:

1. the full InnerLife view detail was about 536 KiB, led by session and digest
   payloads, while processed inbox, ended sessions, and share checks had no
   scheduled retention owner;
2. a deliberately extreme 240-request HTTP burst completed without tool
   errors but blocked the main event loop, with tool p95 around 840 ms and a
   health-probe maximum around 604 ms.

## 2. Scope And Safety Boundary

This version keeps SQLite, the shared HTTP Gateway, and current domain roles.
It does not add a Worker, redesign InnerLife, delete pending work, or turn
retention into a user-operated UI feature.

All destructive retention validation uses an isolated consistent database
copy. The live database is read-only evidence.

## 3. InnerLife Retention

Desktop daily maintenance owns one bounded cleanup with these defaults:

- processed inbox: 30 days, newest 500 rows per Agent;
- ended sessions: 180 days, newest 200 rows per Agent;
- share checks: 30 days, newest 500 rows per Agent;
- digest runs: newest 200 rows per Agent, reusing the existing contract.

Pending or processing inbox rows, active sessions, shares, thoughts, and
delivery actions are protected. Cleanup records policy, before/after counts,
deletion reasons, and does not run automatic `VACUUM`.

Acceptance:

- age and per-Agent caps are both enforced and idempotent;
- protected rows survive;
- imported digest overflow converges during maintenance rather than waiting
  for a future digest;
- backup/import and existing InnerLife behavior remain intact.

## 4. Compact InnerLife Detail

The InnerLife view snapshot uses compact rows containing only fields rendered
by the list: identity, Agent, status/kind, bounded preview, and timestamps.
Existing record/detail repository methods retain full payload access.

Acceptance on the maintained product-scale copy:

- InnerLife view payload is at most 160 KiB, down from about 536 KiB;
- view IPC p95 stays below 50 ms;
- pagination totals and Agent filtering stay correct;
- reading the page remains observational and does not change lifecycle state.

## 5. HTTP Backpressure

The main-process HTTP Gateway owns a small fair admission controller:

- maximum active tool calls: 8;
- maximum queued tool calls: 64;
- bounded queue wait;
- excess or expired requests receive HTTP `429`, JSON-RPC busy evidence, and
  `Retry-After`;
- health, initialize, tools/list, and ping remain outside the tool queue.

Acceptance:

- normal 1/4/8-Agent sequential-per-Agent workloads have zero failures;
- eight-Agent health p95 stays below 50 ms;
- a 240-request simultaneous burst returns explicit backpressure, no HTTP 500,
  and keeps health p95 below 50 ms;
- authentication and Agent/client/conversation identity remain unchanged;
- stop releases queued requests instead of leaving promises alive.

## 6. Validation And Stop Boundary

Run focused retention, compact-payload, HTTP contract, normal concurrency, and
burst tests first. Then run the product-scale runtime and SQLite baselines,
`npm run test:smoke`, and a five-minute long-run check. The 30-minute gate is
still required before a future package candidate is called performance-
validated.

This plan authorizes local code, tests, docs, version updates, and scoped local
commits. It does not authorize push, package, installation, tag, release, or
deployment.

## 7. Implemented Results

- compact synthetic InnerLife detail: 58,906 bytes against the 163,840-byte
  ceiling, with full single-record detail preserved;
- product-scale 45.7 MB copy: runtime snapshot p95 19.6 ms and InnerLife
  navigation 15.2 ms;
- shared HTTP normal workload: 1/4/8-Agent p95 7.1/3.1/4.3 ms, zero failures;
- 240-call HTTP burst: 204 successes, 36 explicit 429 responses, zero HTTP 500,
  health p95 25.6 ms;
- product-scale stdio SQLite workload: 1/4/8-Agent p95 8.0/10.3/18.4 ms,
  zero failures, zero lock failures, and `quick_check=ok`;
- focused simultaneous stdio startup: eight Agents, zero lock failures after
  applying busy timeout before WAL initialization.
- five-minute isolated long run: 60 samples, runner RSS -6.0 MB, Gateway RSS
  -11 MB, and total measured growth +0 B against the 64 MB short-run ceiling.

The maintained evidence still does not authorize a Worker boundary: both the
normal and burst health p95 remain below 50 ms. A future package candidate still
needs the 30-minute long-run gate named above. `npm run test:smoke` and the full
Streamable HTTP Gateway smoke both passed for this local implementation.

## 8. Installed Lite Acceptance

The user-installed macOS arm64 Lite app reported `0.6.2` and passed the final
installed-runtime loop against the running Gateway and a consistent 46.5 MB
real-database copy:

- startup to ready: 491.7 ms;
- steady overview IPC p95: 16.7 ms;
- InnerLife IPC p95: 11.3 ms with a 98,126-byte payload;
- normal 1/4/8-Agent HTTP read p95: 9.5/20.5/34.5 ms, zero failures;
- 240-call burst: 194 successes, 46 explicit 429 responses, zero HTTP 500,
  health p95 25.7 ms;
- five-minute installed endurance: 60 tool calls, zero failures, health p95
  1.8 ms, combined private-memory growth -22.2 MB.

The live database was not destructively cleaned during acceptance. A consistent
copy proved the new retention policy would delete 857 capacity-only InnerLife
history rows while preserving all 87 pending/processing inbox rows and 21
active sessions, with `quick_check=ok`. The next daily maintenance owns live
convergence. The 30-minute release endurance gate remains unrun.
