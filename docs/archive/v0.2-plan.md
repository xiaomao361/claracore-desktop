# ClaraCore Desktop v0.2 Plan

Status: superseded by `product-core-reset.md`.

This document captured the first v0.2 direction where Desktop was still a local control center around existing services. The current direction is different: Desktop is the main product, existing services are references for redesign, and the new product must not interfere with the current local Gateway during development.

## Goal

v0.2 turns ClaraCore Desktop from a read-only shell into the first usable local control center.

The target user flow is:

1. Install and open ClaraCore Desktop on macOS.
2. Point it at existing ClaraCore data or import a copy.
3. Let Desktop start and supervise the Gateway it owns.
4. Give agents a clear MCP setup document generated by Desktop.
5. Use each service's own Web UI for deeper service-specific work, with a shared ClaraCore visual style.

## Product Boundary

Desktop is the installer, control center, and agent handoff surface.

Desktop owns:

- Service lifecycle overview and actions.
- Data directory selection and import/export entry points.
- Gateway ownership and conflict detection.
- Shared model/configuration entry points.
- Agent setup documentation.
- Packaging and bundled runtime.

Service Web UIs own:

- Memoria detailed search, record inspection, and cleanup.
- Continuity detailed thread/current-position inspection.
- InnerLife detailed daemon state, thoughts, and logs.
- Service-specific troubleshooting.

Desktop should not rewrite every deep service operation. It should open those service pages in a consistent ClaraCore style.

## UI Unification

The three service Web UIs should adopt the Desktop look and interaction model.

Shared style requirements:

- Same light local-tool visual language as Desktop.
- Same cards, status badges, tables, buttons, and form density.
- Same bilingual tone where pages expose user-facing controls.
- Same route-level header pattern: summary first, details below.
- Same action pattern: safe read actions visible; destructive actions behind confirmation.

Implementation direction:

- Add a shared stylesheet under the ClaraCore shared layer, for example `shared/ui/claracore.css`.
- Each service Web UI imports that stylesheet.
- Avoid moving service UI code into Desktop unless the service has no Web UI.

## Configuration Surface

v0.2 must expose the two currently hidden configuration areas:

### Memoria Embedding

Desktop should show and eventually edit:

- Provider: built-in, Ollama, OpenAI-compatible.
- Model name.
- Endpoint/base URL.
- API key location, never the secret value by default.
- Current status.
- Test result.

Memoria should own validation and persistence through a service config API or CLI.

### InnerLife Daemon Model

Desktop should show and eventually edit:

- Daemon enabled/disabled.
- Provider: disabled, Ollama, OpenAI-compatible, custom command.
- Model name.
- Endpoint/base URL.
- Loop interval.
- Runtime budget limits.
- Current daemon status.
- Test result.

Desktop Models owns validation and persistence through product settings. The
old InnerLife service remains a reference and import source, not a runtime
configuration authority.

## Agent Setup

Desktop must provide a first-class `Agent Setup` page.

The page should generate a copyable Markdown document containing:

- Recommended Gateway MCP config.
- Gateway command and current Python path.
- Current data directory.
- Desktop-managed HTTP endpoints.
- Sub-service MCP surfaces.
- Sub-service CLI fallback commands.
- Health-check commands.
- Troubleshooting notes for wrong endpoint, port conflict, Python mismatch, and missing packages.

The generated document should also be available as:

- A copy button in Desktop.
- An exportable `claracore-agent-setup.md` file.
- Later, a local read-only URL for agents to fetch.

The recommended agent path is the unified Gateway MCP entry. Direct sub-service MCP should be documented as fallback or advanced use only.

## Python Runtime

Packaged Desktop should be self-contained.

Runtime plan:

- Packaged mode uses bundled Python by default.
- Development mode may use the existing configured Python.
- The current Python source is always visible: bundled, system override, or development checkout.
- Agent setup docs must include the exact Python path used by Gateway.
- Users can override Python for development, but release defaults must not require conda.

## Data Strategy

Desktop should support three data modes:

- Use existing data: use the current `~/.claracore` data in place.
- Import copy: copy existing data into a Desktop-managed directory.
- Empty new core: initialize a fresh Desktop-managed data directory.

Default behavior should be conservative:

- Never move or delete existing data automatically.
- Scan before import.
- Show a preview before copy.
- Make backups before restore.

## Gateway Ownership

Desktop must distinguish:

- Gateway started by Desktop.
- Gateway already running outside Desktop.
- Gateway stopped.
- Port occupied by an unknown process.

Rules:

- Desktop can stop only services it started.
- If an external Gateway is already running, Desktop should show it and avoid taking over silently.
- Gateway startup errors must be visible in the UI.

## v0.2 Scope

Included:

- v0.2 planning and architecture docs.
- Agent Setup page with generated Markdown.
- Desktop Models page for Memoria embedding and InnerLife daemon model configuration.
- First pass at service Web UI style contract.
- Gateway lifecycle design and first implementation step.
- macOS packaging plan.

Not included:

- Windows packaging.
- Cloud sync.
- Full agent registry.
- Built-in chat.
- Rewriting service Web apps inside Desktop.

## Completion Criteria

v0.2 is complete when:

- Desktop can start the Gateway it owns.
- Desktop can detect an externally started Gateway and avoid killing it.
- Desktop shows real service health.
- Desktop shows a Models surface for Memoria embedding and InnerLife daemon model settings.
- Desktop generates a useful agent setup document.
- The service Web UIs have a consistent ClaraCore visual style.
- A macOS package can be built and opened.
- The package can use existing ClaraCore data without modifying it unexpectedly.
