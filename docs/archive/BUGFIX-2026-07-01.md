# Bugfix Notes - 2026-07-01

## Gateway Agent Identity Precedence

### Symptom

Agent Access showed mixed `lara` and `hermes:lara` Gateway traces even after the
Hermes-side MCP configuration had been changed to `lara`.

### Cause

Gateway previously allowed request-level `agentId` or `agent_id` arguments to
win before the process environment. A stale packaged Gateway process can also
keep its launch-time `CLARACORE_AGENT_ID`, so an old process launched with
`hermes:lara` continued to emit old identity rows until it was stopped.

The live July 1 follow-up showed this exact shape: Hermes LaunchAgent
`ai.hermes.gateway` had already loaded the corrected MCP config, but it still
owned an older ClaraCore Desktop `--gateway` child whose command line contained
`--env CLARACORE_AGENT_ID=hermes:lara`. That child emitted an
`innerlife_pending_shares` trace at local time `2026-07-01 09:50:30`.

### Fix

- Gateway now treats `CLARACORE_AGENT_ID` as the authoritative MCP process
  identity.
- Gateway normalizes tool-call arguments before tracing, so stored request JSON
  uses the process identity and removes legacy `agent_id`.
- Agent setup and Gateway docs now recommend stable ids such as `lara`, `clara`,
  and `codex` instead of tool-prefixed examples.
- `agent_identity_merge` now also repairs `gateway_traces.request_json`, not
  only the top-level `agent_id` column.

### Data Repair

Before merging old rows, a product database backup was created:

`~/Library/Application Support/claracore-desktop/data/backups/manual/claracore-before-agent-merge-20260701-084502.db`

The live product database was checked with `PRAGMA quick_check`, then legacy
`hermes:lara` rows were merged into `lara`. A follow-up check confirmed there
were no remaining `gateway_traces` rows where either `agent_id`,
`request_json.agentId`, or `request_json.agent_id` still contained
`hermes:lara`.

### Operational Rule

Changing an agent's configured id is not enough if an old stdio Gateway process
is still running. Restart the agent client, or stop stale packaged
`--gateway` processes, then run `agent_identity_merge` if historical rows need
to be consolidated.

Useful checks when this happens again:

```bash
ps auxww | rg "ClaraCore Desktop --gateway|CLARACORE_AGENT_ID"
launchctl print gui/$(id -u)/ai.hermes.gateway
sqlite3 "$HOME/Library/Application Support/claracore-desktop/data/claracore.db" \
  "select agent_id, tool_name, request_json, datetime(created_at,'+8 hours') from gateway_traces order by created_at desc limit 10;"
```

### Follow-up Hardening

`agent_identity_merge` now guards JSON lookups with `json_valid()` for Gateway
trace requests and Shared Line position metadata. This keeps historical malformed
JSON rows from aborting identity repair with SQLite `malformed JSON` errors.

A live follow-up merge also consolidated the remaining single `hermes:lara`
Gateway trace into `lara` after creating this backup:

`~/Library/Application Support/claracore-desktop/data/backups/manual/claracore-before-agent-merge-hardening-20260701-094734.db`

Follow-up verification:

- `PRAGMA quick_check`: `ok`
- malformed Gateway trace request JSON rows: `0`
- malformed Shared Line position metadata rows: `0`
- remaining `hermes:lara` Gateway trace rows or request JSON references: `0`
