# ClaraCore Desktop v0.6.1 Performance Plan

> Status: implementation and local performance validation complete
> Target version: `0.6.1`
> Public release boundary: none; current public release remains `0.5.8`

## 1. Objective

v0.6.1 reduces measured Desktop and multi-Agent runtime overhead without
replacing SQLite, weakening Agent auditability, or coupling Memoria,
Shared Line, and InnerLife into one domain.

The work proceeds in order. Each phase must meet its own functional and
performance acceptance checks before the next phase starts. Every phase gets a
separate local commit. Push, packaging, release publication, and deployment are
outside this plan unless separately authorized.

## 2. Current Evidence

The v0.6.0 baseline and the follow-up read-only diagnosis established:

- a full runtime snapshot against a consistent copy of the 43.4 MB product
  database issued about 95 SQL calls and serialized about 936 KB;
- InnerLife contributed about 528 KB and Shared Line about 251 KB to that
  snapshot;
- Logs Follow rebuilt the full snapshot every two seconds;
- `gateway_context` loaded full InnerLife and non-lite Shared Line state before
  returning a bounded context packet, and repeated the doctor read;
- Memory create/update waited for embedding work before acknowledging the
  durable write;
- InnerLife Session End waited for afterthought generation in the request path;
- one stdio Gateway process used about 57 MiB RSS, with eight processes
  reporting about 455.5 MiB aggregate RSS;
- the maintained 1/4/8-Agent SQLite workload completed with zero lock failures
  and post-change p95 latency of 3.9/7.1/17.3 ms;
- Gateway startup stayed around 82-91 ms through eight simultaneous processes.

SQLite replacement and a broad renderer rewrite are therefore not justified.
The first target is query/payload amplification, followed by synchronous long
work and connection ownership.

## 3. Phase 1: Snapshot And Logs

### Solution

Split the current global snapshot into maintained capability surfaces:

1. a lightweight runtime overview for shell, Home, Settings, and Agent Access;
2. a bounded Logs snapshot for runtime events, Gateway traces, decay status,
   and the summaries required by Time Flow;
3. existing or focused detail IPC methods for Memory, Shared Line, InnerLife,
   Trace, and Data pages;
4. renderer-side view hydration with one in-flight request per view and scoped
   invalidation after writes.

Do not introduce a generic client state framework. Reuse the existing view
modules and IPC contracts. Preserve read-only fingerprints for observation
paths.

### Acceptance

- runtime overview: at most 30 SQL calls and at most 200 KB on the maintained
  43.4 MB data copy;
- Logs refresh: at most 10 SQL calls and at most 100 KB;
- Logs Follow no longer calls the full runtime snapshot every two seconds;
- Home, Settings, Agent Access, Logs, Shared Line, and InnerLife render their
  existing semantics;
- a missing or slow detail request produces a bounded loading/error state, not
  a blank or stale whole application;
- observation reads do not add domain rows.

## 4. Phase 2: Gateway Context

### Solution

Keep the `gateway_context` response contract while changing its inputs to:

- `getResumePacket(..., lite: true)` for the selected Shared Line;
- lite InnerLife counts, daemon state, profiles, and doctor state;
- targeted pending inbox and recent-share queries for the calling Agent;
- one doctor calculation only;
- explicit limits before JSON mapping rather than full-snapshot filtering.

Semantic Memory query behavior remains independent and keeps its own measured
budget.

### Acceptance

- response keys and guidance text remain backward compatible;
- the no-query path has p95 at or below 100 ms on the maintained data copy;
- SQL calls fall by at least 50 percent from the pre-change no-query path;
- no cross-Agent Inbox or share content enters the packet;
- ambiguous Shared Line selection remains a safe, explicit result.

## 5. Phase 3: Durable Background Work

### Memory solution

Memory create/update commits the Memory and a persisted pending embedding state
before returning. A Desktop-owned worker drains pending embeddings with bounded
concurrency, retry/backoff metadata, and restart recovery. Keyword retrieval
must see the Memory immediately; semantic retrieval becomes available after the
embedding succeeds.

### InnerLife solution

Session End commits the closed session and a persisted afterthought job before
returning. The InnerLife owner processes the job idempotently and records the
generated or fallback thought/share. No untracked promise may outlive a Gateway
request.

### Acceptance

- Memory create/update durable acknowledgement p95 is at or below 100 ms when
  model execution is excluded;
- Session End durable acknowledgement p95 is at or below 200 ms;
- pending work survives process exit and completes after restart;
- retry exhaustion is visible through maintained status/doctor surfaces;
- duplicate workers cannot create duplicate embeddings, thoughts, or shares;
- existing keyword, archive, backup, import, and Agent identity behavior stays
  intact.

## 6. Phase 4: Multi-Agent Connection Ownership

### Solution

Use the existing shared Streamable HTTP endpoint as the preferred first-party
connection and retain stdio as a compatibility fallback. Add a combined
Electron UI plus 1/4/8-Agent HTTP profile that measures event-loop delay,
navigation latency, Gateway latency, RSS, and lock failures.

If the measured main-thread budget fails, move HTTP tool execution and its
synchronous `node:sqlite` connection into a maintained Worker boundary. Do not
add that boundary without failing evidence; the Worker must have explicit
startup, error, restart, and quit ownership.

### Acceptance

- first-party setup continues to prefer shared HTTP and truthfully documents
  stdio process cost;
- eight HTTP Agents do not create eight Gateway processes;
- UI navigation remains below the maintained common-action threshold while the
  concurrent Gateway workload runs;
- event-loop p95 remains below 50 ms; otherwise the Worker boundary is required
  and revalidated;
- Gateway authentication, stable Agent identity, conversation identity, and
  local-only binding remain unchanged.

## 7. Phase 5: Gateway Trace Write And Retention

### Solution

- remove the unused trace read-back after insert while keeping the trace write
  awaited before the tool response completes;
- retain successful traces for 30 days with a newest-10,000 cap;
- retain error traces for 180 days while protecting the newest 2,000 errors;
- enforce a total hard cap of 20,000 traces;
- run cleanup from the Desktop-owned daily maintenance cadence;
- record a bounded runtime-event receipt with policy, before/after totals, and
  deletion counts;
- do not run automatic `VACUUM`; use passive checkpoint evidence separately if
  WAL behavior needs diagnosis.

### Acceptance

- trace list/filter and Agent activity summaries retain their semantics;
- cleanup preserves protected recent errors and obeys all caps;
- cleanup is idempotent and tested with multiple Agent identities;
- concurrent Gateway baseline still has zero lock failures;
- the live database is never used as the write-load test target.

## 8. Phase 6: Integrated Validation

Run the maintained checks in proportion to the touched boundaries:

- full static and contract checks;
- Home, Logs, Shared Line, InnerLife, Trace, Settings, and Agent Access Electron
  smokes;
- Gateway stdio and Streamable HTTP contracts;
- Memory, Controller, backup/import, and retention tests;
- runtime snapshot and multi-Agent SQLite baselines;
- combined UI plus HTTP Agent profile;
- full 30-minute memory/endurance gate before a package candidate is called
  performance-validated.

Update the v0.6.1 measured results beside the v0.6.0 baseline. Do not overwrite
the v0.6.0 evidence or rename its Memory Controller plan.

## 9. Commit And Stop Boundaries

Planned local commits:

1. version and frozen performance plan;
2. snapshot/Logs split;
3. Gateway context targeting;
4. durable background work;
5. multi-Agent connection boundary;
6. trace retention;
7. integrated docs and validation closeout.

Each commit reports what was validated and what remains unverified. The plan
stops at local validation and commits by default. It does not imply push,
package, tag, GitHub Release, installation, or deployment.

## 10. v0.6.1 Measured Results

Completed locally on 2026-07-21:

- the maintained 44.1 MB consistent database copy now builds the runtime
  overview with 30 SQL reads and a 157,166-byte payload, down from about 95
  reads and 936 KB for the v0.6.0 full snapshot;
- the focused Logs refresh uses two SQL reads; the fixture payload is 493
  bytes, and Follow no longer rebuilds the runtime overview;
- `gateway_context` uses at most 23 SQL reads in the maintained no-query smoke,
  down from the 51-read reference path (54.9 percent reduction), with a 0.777
  ms p95 across 20 local samples and explicit cross-Agent isolation checks;
- Memory writes return a persisted pending embedding state instead of waiting
  for model work. Session End persisted its afterthought job in 0.505 ms in the
  restart-recovery smoke; two concurrent workers claimed the job only once;
- the shared HTTP/UI profile completed 30, 120, and 240 calls for 1, 4, and 8
  Agents with zero failures. At eight Agents, main-event-loop health p95 was
  1.6 ms and UI IPC p95 was 22.8 ms, so the 50 ms Worker threshold did not fail;
- trace insert performs zero SQL read-back queries, requests above 16 KiB are
  bounded, and the retention fixture preserved protected errors while applying
  age and capacity rules without `VACUUM`;
- the product-scale runtime snapshot p95 was 16.1 ms across 20 samples;
- the product-scale stdio SQLite profile completed 1/4/8-Agent workloads with
  zero failures and zero lock failures; overall p95 was 4.0/9.0/16.1 ms.

The static/contract/UI smoke suite is required to pass after the final docs
commit. The 30-minute endurance gate, packaging, signing, installation, push,
tag, release publication, and deployment remain outside this local
implementation closeout; the endurance gate is required before calling a
future package candidate performance-validated.
