# ClaraCore Desktop v0.6.0 Memory Controller Plan

> Status: Slice 0-1 observe-only checkpoint validated; injection canary and stable release gates pending
> Target version: `0.6.0`
> Decision date: 2026-07-20
> Review revision: 2026-07-21

## 1. Outcome

ClaraCore Desktop v0.6.0 adds an agent-first Memory Controller above Memoria.
It decides whether memory should participate in the current turn, which bounded
memory context is safe to inject, and how the result should be observed and
evaluated.

```text
Memoria: durable facts, relations, lifecycle truth
                     ↓
Memory Controller: NOOP / RETRIEVE / ABSTAIN / INJECT / EXPAND
                     ↓
Clara / Lara / Codex / InnerLife
```

The controller is not a new memory store and does not move retrieval policy into
Memoria. Memoria remains independently usable through its current MCP, CLI,
database, graph, archive, supersede, and maintenance contracts.

The v0.6.0 release is successful when fixed per-turn retrieval has been replaced
by a bounded, observable decision path; relevant memories can reach an Agent
without flooding context; and the system records enough trustworthy feedback to
evaluate a future learned policy.

### 1.1 What We Are Building

v0.6.0 does five concrete things:

1. Build and validate a deterministic Stage A gate that decides `NOOP` or
   `RETRIEVE` without calling an LLM or semantic search.
2. Add a Desktop-owned controller and append-only event ledger that records why
   each decision was made.
3. Add a bounded, invalidatable retrieval cache so repeated prompts do not pay
   the full semantic-search cost.
4. Canary high-confidence Memory injection through creflex, with auditable
   response evidence and a kill switch.
5. Add bounded one-hop relation expansion, then close the release across
   migration, backup/import/export, Full/Lite, documentation, and packaging.

v0.6.0 does **not** implement a contextual bandit. The event and feedback ledger
is the product output that makes an evidence-backed v0.6.1 learning experiment
possible after enough real usage accumulates.

Estimated implementation size: four to five weeks for Slices 0-3 plus release
closeout, assuming one bounded slice is completed and validated at a time.

Work proceeds in this order:

| Order | Deliverable | Decision unlocked |
| --- | --- | --- |
| 0 | Stage A rules, multilingual fixtures, raw confusion matrix, latency baseline | Whether deterministic recall triggers are good enough for a canary |
| 1 | Controller contract, event ledger, retention, safe cache, observe-only Gateway path | Whether decisions are safe, bounded, and auditable without injecting context |
| 2 | creflex top-1 canary, evidence-backed feedback, kill switch, raw outcome report | Whether controlled Memory injection is useful enough to keep enabled |
| 3 | Typed one-hop expansion and explicit-stuck re-retrieval | Whether relation-aware recall adds value without loops or scope leakage |
| 4 | Migration, backup/import/export, Full/Lite, docs, packaging, release checks | Whether v0.6.0 is releasable |

## 2. Why This Version Exists

The current product has crossed the point where one fixed retrieval rule is
adequate:

- the live library already contains hundreds of active Memories and dozens of
  explicit relationship edges;
- `memoria_search` supports semantic recall, current/historical time views, and
  one-hop related results;
- knowledge-card pointers proved that useful material can be stored without
  being naturally recalled later;
- `post-memoria-create.js` supplies write-time related-memory candidates, but
  the read/conversation half of the loop does not exist;
- `user-prompt-submit.js` currently injects time and a pending-share count only;
- a 2026-07-20 live trace sample showed 81 `memoria_search` calls over seven
  days, about 1235 ms average duration, and a maximum near 3 seconds. Calling
  semantic search on every message would make latency and context noise part of
  the default path.

The product direction already says that ClaraCore should maintain connections
that are explainable, strengthenable, and weakenable. v0.6.0 turns retrieval
from a fixed hook into the first controlled use of that network.

## 3. Product Principles

1. **Agent first.** Memory context is delivered to the Agent that can use it;
   v0.6.0 does not add a new human dashboard.
2. **Storage and control stay separate.** Memoria owns facts and relations. The
   controller owns access decisions and their feedback.
3. **Abstention is a real action.** Injecting a wrong memory is more harmful
   than missing a weakly relevant one.
4. **Semantic read paths stay semantically read-only.** A controller decision
   must not create or modify Memory, labels, links, Shared Line, InnerLife, or
   default records. Controller-owned operational decisions and feedback may be
   appended to controller audit tables.
5. **No hidden semantic writes.** Hooks and policies may read, select, inject,
   and record operational evidence. Only an explicit product or Agent write may
   change durable semantic state.
6. **Fail open.** Controller timeout or failure must not block the user's
   message or the Agent runtime.
7. **Bounded context.** Candidate count, graph expansion, text length, latency,
   and injected tokens all have hard caps.
8. **Collect evidence before learning.** v0.6.0 produces trustworthy controller
   events and feedback. Learned-policy code starts in v0.6.1 at the earliest.

## 4. v0.6.0 Scope

### 4.1 Two-stage decision path

The controller must not run semantic search merely to decide whether semantic
search is necessary. It therefore uses two stages.

#### Stage A: pre-retrieval gate

Inputs are cheap turn and runtime facts, for example:

- stable Agent, client, conversation, and session identity;
- whether the prompt explicitly asks about history, a previous decision,
  continuation, preference, or prior knowledge;
- task phase when the harness can provide it: start, progressing, stuck, or
  closing;
- prompt length/category and whether the turn is a likely real-time lookup;
- recent controller outcomes for the same Agent and intent class;
- available context budget and controller mode.

The v0.6.0 gate is deterministic. It may use normalized multilingual phrases,
regular expressions, explicit harness metadata, recent controller state, and
negative rules. It must not call an LLM, calculate an embedding, or call
`memoria_search` while deciding whether retrieval is needed.

Initial positive families include:

- history: `上次`, `之前`, `当时`, `还记得吗`, `你说过`, `last time`,
  `earlier`, `do you remember`;
- continuation: `继续刚才`, `接着上次`, `回到那个`, `continue from`,
  `pick up where we left off`;
- prior decision/preference: `我们怎么定的`, `之前决定`, `我的偏好`,
  `what did we decide`, `my previous preference`;
- knowledge reuse: `之前那个方法`, `那张卡片`, `以前踩过这个坑吗`,
  `reuse the earlier approach`.

The rule set must also contain negative and ambiguous fixtures such as:

- `记得提交代码` or `remember to commit`, which is an instruction rather than
  a history request;
- `之前先执行 A 再执行 B`, which can describe an order inside the current
  task;
- `继续运行当前命令`, which is current execution rather than cross-turn
  continuity;
- `不要回忆旧方案`, which must not trigger injection.

Stage A is evaluated independently before Stage B implementation. Its fixture
must report the raw confusion matrix for Chinese, English, mixed-language,
colloquial, elliptical, negative, and near-miss prompts. A sophisticated Stage
B cannot compensate for a missed Stage A retrieval trigger.

Actions:

- `NOOP`: do not call Memoria;
- `RETRIEVE`: call bounded `memoria_search` with controller-owned parameters.

The deterministic baseline defaults ordinary present-tense turns to `NOOP` and
uses `RETRIEVE` for explicit history, continuation, prior-decision, stable
preference, and knowledge-reuse signals.

#### Stage B: post-retrieval selection

Inputs include:

- top score and score margin;
- result source and current/historical status;
- sensitivity and eligibility;
- relationship kind and one-hop availability;
- prior feedback for a candidate or intent class;
- estimated injected token count.

Actions:

- `ABSTAIN`: results are too weak, ambiguous, stale, restricted, or costly;
- `INJECT_TOP1`: inject one high-confidence candidate;
- `INJECT_TOPK`: inject a bounded set, never more than three;
- `EXPAND_ONE_HOP`: add only a bounded, typed relation neighborhood;
- `RE_RETRIEVE`: reserved behind an explicit stuck-state input; not used on
  every ordinary turn.

### 4.2 Rule policy in production

v0.6.0 ships a human-readable deterministic policy as the production policy.
It must expose a policy version and a reason code for every decision.

Initial reason-code examples:

- `explicit_history_request`;
- `continuation_request`;
- `stable_preference_request`;
- `prior_decision_request`;
- `reusable_knowledge_request`;
- `ordinary_current_turn`;
- `low_relevance`;
- `ambiguous_top_results`;
- `restricted_result`;
- `context_budget_exceeded`;
- `controller_timeout`.

Thresholds and top-k values belong to the controller policy, not to each hook or
client integration.

### 4.3 Auditable event and feedback ledger

Add controller-owned persistence separate from `gateway_traces`.

`memory_control_events` records one decision:

- decision id and policy version/mode;
- Agent, client, conversation, session, and timestamp;
- bounded feature/state JSON;
- Stage A and Stage B actions and reason codes;
- search parameters, candidate ids/scores, and injected ids;
- search and total controller latency;
- estimated injected tokens;
- result status: completed, abstained, timeout, or error.

Do not persist the full user prompt by default. Store a normalized query hash
and, only when diagnostics require it, a short bounded preview that follows the
existing local sensitivity boundary.

`memory_control_feedback` records evidence attached to a decision:

- `used`: the Agent materially relied on an injected Memory;
- `ignored`: context was delivered but not used;
- `wrong`: the Memory was not applicable or caused a wrong direction;
- `corrected`: the user or later evidence corrected the injected Memory;
- `task_succeeded` / `task_failed` / `outcome_unknown`;
- feedback source and bounded evidence metadata.

Agent self-report is a weak signal. User correction and verified task outcome
are stronger signals and must remain distinguishable. Absence of feedback is
`unknown`, not success.

Feedback follows the v0.5.7 InnerLife delivery-evidence precedent rather than
trusting an Agent to mark a result without proof. A `used` report must include:

- decision id;
- used Memory ids;
- conversation id;
- optional response id;
- a bounded excerpt from the actual response or action receipt;
- observed/shared timestamp;
- evidence source.

The server verifies that every used Memory id was actually injected by that
decision, caller and conversation identity match, the evidence time is not
before the decision, and the excerpt passes the maintained minimum. Duplicate
feedback is idempotent. `wrong`, `corrected`, `task_succeeded`, and
`task_failed` accept stronger user-correction or validation receipts without
collapsing them into Agent self-report.

The ledger distinguishes:

- `injected`: controller placed context into the Agent turn;
- `delivered`: a response receipt proves the context reached the response;
- `used`: evidence shows the response or action materially relied on it;
- `task_succeeded` / `task_failed`: later validation proves the outcome.

Canary integrations should automatically record the lowest trustworthy delivery
receipt at response completion when the host exposes that lifecycle point.
Semantic `used`/`wrong` feedback may still be explicit. Missing feedback stays
`unknown`; it is never omitted from reporting merely because successful Agents
were more likely to report.

### 4.4 Gateway and harness boundary

The controller belongs in the Desktop core, not inside a Clara-only hook.

Planned core surface:

- `core/memory-controller/`: feature extraction, deterministic policy,
  retrieval selection, cache ownership, and context formatting;
- `core/db/repositories/memory-controller.js`: events, feedback, retention,
  and bounded queries;
- Gateway handler/definition for an Agent-facing `memory_context` operation;
- Gateway handler/definition for `memory_context_feedback`;
- isolated tests under `core/tests/`.

`memory_context` derives caller identity from the existing Gateway context. It
must not trust a body-supplied Agent id over the authenticated caller. Its
response is a compact packet containing:

- decision id;
- action and reason;
- zero or more bounded Memory candidates;
- one injection-ready context string;
- policy mode/version and timing;
- no mutation instructions.

Harnesses such as creflex may call this operation and, only in a later enabled
canary mode, inject its returned context. They must not implement independent
thresholds or ranking rules. Claude/creflex is the first integration and now
calls the operation in observe-only mode; it deliberately ignores the returned
packet. The Gateway contract remains usable by Hermes, Codex, and future
clients.

### 4.5 Bounded retrieval cache

Caching is part of Slice 1, not a later optimization. creflex
`UserPromptSubmit` is synchronous, so a retrieval decision adds the full search
duration to that user turn.

The cache belongs to the Desktop controller core, not to creflex. A cache key
must include at least:

- normalized query hash;
- Agent/sensitivity scope;
- `timeView`;
- policy version;
- retrieval parameters;
- a Memoria mutation/embedding watermark.

The cache has a short TTL, hard entry/byte cap, hit/miss telemetry, and explicit
invalidation for Memory create/update/restrict/archive/supersede/delete,
relevant label/link changes, and embedding refresh. A hit must revalidate that
candidate ids remain eligible before injection.

Every user turn still receives a new decision id and event. A cache hit records
its source/cache metadata but never reuses a prior decision id, because feedback
must refer to the current injection.

### 4.6 Existing lifecycle boundary

v0.6.0 does not generate consolidation or forgetting proposals and does not
change durable Memory lifecycle state. Existing explicit Memoria operations
remain the only path for merge, archive, supersede, restriction, or deletion.
Controller evidence may support a separately reviewed future proposal workflow,
but it is not interpreted as permission to mutate Memory.

## 5. Explicit Non-Goals

v0.6.0 does not include:

- a new Memory Controller dashboard or engagement surface;
- a fourth semantic storage system;
- automatic knowledge-card generation or a Knowledge Workspace;
- the deferred Experience/procedural-memory layer;
- automatic plan-library generation or plan injection;
- autonomous Memory merging, archiving, superseding, or deletion;
- unrestricted multi-hop graph traversal;
- hidden LLM calls for controller decisions;
- contextual-bandit, learned-policy, replay, or exploration code;
- automatic stuck detection; v0.6.0 accepts only an explicit trusted stuck
  signal;
- replacing explicit `memoria_search` for research, maintenance, audit, or
  historical comparison;
- a full chat client or typing/idle detection.

## 6. Data Migration And Compatibility

- Additive schema migrations only. Existing Memory, link, label, embedding,
  Gateway trace, Shared Line, and InnerLife rows remain unchanged.
- Empty controller tables are a valid upgraded state; no backfill from old
  Gateway traces is required because old traces cannot prove usage or outcome.
- Controller events need a retention policy from their first release: maximum
  age and row/byte cap, feedback-aware deletion behavior, aggregate preservation
  where useful, manual cleanup, and backup/import/export ownership.
- Full and Lite builds use the same controller contract. Embedding-provider
  differences must not change policy semantics.
- Import/export and backup coverage must include controller tables before the
  version is released.
- A controller-disabled instance behaves like v0.5.x: Agents can still call
  Memoria explicitly.
- Failure, timeout, or schema incompatibility degrades to `NOOP`, never to broad
  retrieval.

## 7. Performance And Context Budgets

Initial hard budgets:

- Stage A `NOOP` path: no network embedding/search work and less than 20 ms p95;
- retrieve path: 2500 ms hard timeout, with a release target at or below the
  measured existing search latency rather than adding another model call;
- repeated equivalent retrievals use the bounded controller cache from Slice 1;
- default injection: one Memory;
- hard candidate cap: three Memories;
- graph expansion: one hop and within the same total candidate cap;
- injected context: target at most 600 tokens, hard cap 900 tokens;
- no full body duplication when a compact title/body preview and Memory id are
  enough for the Agent to decide whether to fetch more;
- all controller queries and lists are paged or bounded;
- no controller operation joins the global Home snapshot hot path.

The first retrieval for a new query may still add the measured search latency
to a blocking `UserPromptSubmit` turn. Stage A precision is therefore the main
latency control; caching reduces repeats but cannot justify broad retrieval.
Do not hide latency with an unbounded background queue.

### Slice 0: baseline and fixtures

- Capture a reproducible latency/token baseline from isolated and live-safe
  reads.
- Build an isolated evaluation fixture with positive history/continuation
  prompts, near misses, ordinary negative turns, stale facts, conflicting

- Include existing knowledge-card pointer scenarios as real-shaped fixtures.
- Define the initial multilingual Stage A rules and negative rules.
- Run Stage A alone and report raw true-positive, false-positive,

Done when the fixed always-retrieve and explicit-search baselines can be

### Slice 1: controller skeleton and event ledger

- Add migrations and repositories.
- Implement Stage A/Stage B types, deterministic reason codes, bounded packet
  formatting, timeout, and fail-open behavior.
- Implement the bounded cache, Memoria invalidation watermark, and cache
  telemetry.
- Add `memory_context` in observe-only mode: it records the decision but
  returns no injectable content.

Done when isolated Gateway tests prove `NOOP`, retrieval, abstention, identity,
restricted-memory, timeout, semantic read-only behavior, cache hit/invalidation,

### Slice 2: deterministic canary injection

- Integrate creflex as the first canary without moving policy into the hook.
- Enable injection only for selected Agents/intents and high-confidence top-1
  results.
- Add `memory_context_feedback`, delivery-evidence validation, idempotency, and
  response-completion receipt handling where the host supports it.
- Preserve a kill switch and policy-mode setting.

Done when at least 30 real-shaped decisions have been observed, at least 10
retrieval decisions have explicit evidence or remain visibly `unknown`, raw
outcome counts are reported without percentage claims, and no restricted or
wrong-scope Memory crossed the injection boundary.

### Slice 3: relation expansion and stuck re-retrieval

- Allow bounded one-hop expansion when the primary hit and typed relationship
  justify it.
- Allow one alternative re-retrieval only when the harness explicitly reports a
  trusted stuck state. v0.6.0 does not infer stuck from conversation text or
  repeated actions.
- Prove that repeated retrieval cannot loop or repeatedly inject the same ids.

Done when linked retrieval improves fixture coverage without increasing wrong
injection beyond the release gate.

### Slice 4: v0.6.0 release closeout

- Include controller tables in backup/import/export and maintenance checks.
- Update architecture, code map, MCP playbook, runtime policy, release notes,
  and package metadata.
- Validate source, Gateway, isolated runtime, Full/Lite behavior, package
  boundaries, migration from the public version, and controller-disabled
  fallback.
- Validate event retention/cleanup and produce a raw canary report showing
  injected, delivered, used, ignored, wrong, corrected, and unknown counts.
- Package and publish only through the normal explicit release-closeout flow.

## 10. Release Gates

v0.6.0 is not ready merely because the controller returns results.

Required gates:

1. **Safety:** zero restricted-memory injection and zero semantic writes from
   controller read paths.
2. **No high-risk false positive:** no wrong-scope, restricted, or otherwise
   high-risk Memory injection is accepted as harmless.
3. **Raw canary truth:** report counts for injected, delivered, used, ignored,
   wrong, corrected, task outcome, and unknown. Most evidenced injections should
   be used, with no repeated wrong-injection pattern; v0.6.0 does not claim a
   statistically meaningful success percentage from the initial canary.
4. **Abstention:** negative and near-miss fixtures reliably produce `NOOP` or
   `ABSTAIN`.
5. **Latency:** ordinary `NOOP` turns add no search latency; retrieval remains
   inside the declared timeout and does not add an extra LLM call.
6. **Context:** no decision exceeds candidate, hop, or token budgets.
7. **Audit:** every injection maps to one decision record and can accept later
   evidence; cache hits still create a new decision record.
8. **Compatibility:** upgrade, backup/restore, Full/Lite, controller-disabled,
   and external Gateway paths pass.
9. **Scope boundary:** v0.6.0 contains no Bandit, learned-policy, replay, or
   exploration implementation.

If the controller cannot meet precision or latency gates, release the event and
feedback foundation with injection disabled rather than relaxing safety.

## 11. Stop Conditions And Rollback

Pause or disable canary injection when any of these occurs:

- a restricted or wrong-scope Memory is selected;
- wrong-memory injection changes an Agent's action in a harmful direction;
- p95 controller latency exceeds the agreed budget for two consecutive canary
  windows;
- repeated context injection produces visible prompt noise;
- feedback cannot be tied to a unique decision;
- the controller mutates semantic state from an ordinary read;
- cache invalidation allows stale, restricted, archived, superseded, or deleted
  Memory to cross the injection boundary.

Rollback is configuration-first: set controller mode to `off` or `observe`.
Existing Memoria tools and data remain available. Additive controller tables may
remain for audit; rollback does not require deleting them.

## 12. Deferred Post-v0.6 Work

Only after v0.6.0 produces trustworthy feedback should a later version consider:

- v0.6.1 offline replay and a contextual-bandit shadow policy, after hundreds
  of trustworthy feedback events or another explicitly reviewed evidence bar;
- promoting any learned retrieval policy from shadow to a bounded canary;
- automatic stuck detection and learned re-retrieval timing;
- task-plan reuse or `PLAN_INJECT`;
- feedback-driven relation strengthening/weakening;
- reviewable consolidation proposals;
- reviewable archive/supersede proposals;
- cross-client policy specialization;
- procedural Experience retrieval;
- a human diagnostics surface, if operational evidence proves it is needed.

## 13. Research References And Claim Boundary

- [Memory as a Controlled Process: Learned Adaptive Memory Management for LLM Agents](https://arxiv.org/abs/2607.13591)
  motivates the backend-independent control layer, abstention, bounded actions,
  and a future lightweight learned policy. Its reported gains come from a
  preprint and structured benchmark feedback; they are not ClaraCore acceptance
  targets, and its learning implementation is deferred to v0.6.1 or later.
- [Oracle AI Agent Memory](https://blogs.oracle.com/developers/oracle-ai-agent-memory-a-governed-unified-memory-core-for-enterprise-ai-agents)
  supports full lifecycle, scoped identity, audit, bounded context, and
  governed deletion as architecture concerns. Its published accuracy and token
  results are vendor-reported under its own model and retrieval configuration.

ClaraCore v0.6.0 must be justified by its own live-safe canary evidence, not by
copying either report's benchmark numbers.

## 14. Implementation Checkpoint: Bounded Cache And Watermark

The second Slice 1 foundation loop now includes:

- an in-process retrieval-candidate cache with deterministic scoped keys, TTL,
  entry-count, byte-size, and three-candidate bounds;
- an additive `005_memory_controller_watermark` migration whose triggers advance
  a singleton revision after Memory, label/alias, link, or embedding mutations;
- mandatory database eligibility revalidation before any candidate-bearing
  cache hit is returned;
- safe misses for expired, missing, restricted, archived,
  superseded, deleted, wrong-agent, or wrong-time-view candidates;
- watermark changes produce new cache keys; superseded entries remain bounded
  by the existing TTL/LRU limits rather than a second invalidation mechanism;
- explicit rejection of decision ids in cache values, preserving the contract
  that every future cache hit receives a new ledger decision event;
- focused smoke coverage using only an isolated temporary database.

The next Slice 1 core loop added deterministic orchestration and Stage B:

- Stage A `NOOP` records a decision without calling Memoria;
- retrieval requires a trusted Agent id and preserves Agent/time/sensitivity
  scope through search, cache, and eligibility revalidation;
- conservative Stage B rules abstain on keyword-only, weak, ambiguous,
  restricted, ineligible, or over-budget results; hybrid keyword+vector hits
  must pass the same cosine and margin gates as vector-only hits;
- the formatter defaults to one Memory and 600 estimated tokens, while honoring
  explicit caller budgets up to the 900-token hard cap;
- observe mode records the prospective Stage B action but returns no context or
  injected ids; every cache hit creates a fresh decision event;
- the retrieve path has a 2500 ms maximum timeout and timeout/search/audit
  failures degrade to a context-free `NOOP`;
- isolated tests cover Codex/Lara scope separation and prove the orchestration
  path does not mutate Memory, Shared Line, or InnerLife semantic tables.

The authenticated observe-only `memory_context` Gateway operation is now
available over stdio and Streamable HTTP. Its schema does not accept an Agent id;
the handler binds the decision to transport identity, unidentified callers fail
closed to a context-free `NOOP`, and body identity spoofing is ignored. Isolated
tests cover NOOP, retrieval, abstention, restricted exclusion, cache-hit decision
uniqueness, timeout, semantic read-only behavior, and both Gateway transports.

This still does not inject context. Clara/Claude Code and Codex are now routed
through the controller on each non-empty `UserPromptSubmit`, while explicit
Memoria operations, Clara's post-create association hook, Codex selective
Shared Line reads, and InnerLife share checks remain separate. Host smokes prove
authenticated prompt/session routing, absence of Controller-result injection,
and Clara fail-open behavior. Hermes/Lara currently expose MCP capability but
no verified per-prompt lifecycle surface, so there is no hidden automatic loop
to route for those hosts. They must adopt `memory_context` when such a surface
is introduced.

The first test-package safety controls are also implemented. Automatic recall
is off by default, the operator can switch only between `off` and `observe` in
Settings > Advanced, and Trace > Advanced data shows bounded raw counts and the
latest ten decisions. Turning the controller off preserves historical evidence
but causes subsequent `memory_context` calls to return `controller_disabled`

export/import now preserve controller settings, watermarks, events, and
feedback. Source-checkout Full/Lite and UI validation pass. The macOS arm64 Lite

## 16. Canary Safety Hardening (2026-07-21)

A pre-canary review found and closed the remaining deterministic-policy and
runtime-isolation gaps while preserving the observe-only product boundary:

- keyword-only results no longer qualify for injection, and hybrid results must
  pass the vector score and margin gates;
- capacity cleanup never deletes feedback-bearing decisions, ledger scans and
  deletes are batched at 500 rows, and regression coverage exercises 1,100
  capacity candidates;
- normalized cache identity now matches the actual retrieval input, watermark
  invalidation has one key-based mechanism, and LRU eviction has a defensive
  empty-map stop;
- afterthought batch failure is isolated from every Agent daemon tick;
- overview `recentShares` uses recent shares rather than pending shares;
- full-width CJK punctuation participates in context-token estimates, and an
  explicit caller budget may use the range above the 600-token default through
  the 900-token hard cap;
- Streamable HTTP applies a per-Agent active-call cap below the global cap;
- an invalid persisted Controller mode returns `invalid_controller_mode`
  instead of silently becoming `off`.

These changes do not expose canary in Settings or Gateway. `off` and `observe`
remain the only product modes until the explicit canary slice begins.
