## #1 — Product JSON import lost/served stale data (WAL corruption class)

**Severity:** high (data loss + stale reads)

**Symptom**
`backup-restore-smoke` failed at the JSON import step: an imported record
(`before`) went missing, and a record that should have been replaced
(`afterC`) survived the import.

**Root cause**
The runtime keeps a long-lived cached SQLite connection
(`ProductDatabase.connection`); `exec`/`query` no longer open and close a
connection per call. That removed the implicit WAL checkpoint that the old
per-call `close()` provided. As a result, in `importProductDataJson`:

1. `tempDatabase` inserts stayed in `temp.db-wal`. `fs.copyFile(tempPath, ...)`
   copies only the main `.db` file, so the uncheckpointed rows were never
   copied — the imported `before` row was lost.
2. The copy overwrote the destination database file while the cached
   connection was still open on it. That connection kept serving rows from its
   own stale `-wal`, so the post-import read still returned `afterC`.

**Fix** (`core/runtime/imports.js`, `importProductDataJson`)
Before `fs.copyFile`:
- `tempDatabase.close()` — checkpoints the temp WAL into the temp `.db` so the
  copy is complete.
- `resetCachedDatabase()` — closes the live destination connection.
- `removeSqliteSidecars(paths.databasePath)` before and after the copy — drops
  `-wal`/`-shm` so the swapped file is read cleanly, never through a stale
  connection's WAL.

Supporting wiring:
- `core/runtime/backup.js` now exports `removeSqliteSidecars` (deletes
  `<db>-wal` and `<db>-shm`, ENOENT tolerated).
- `core/runtime/imports.js` requires it; `createImportRuntime` receives
  `resetCachedDatabase`.
- `core/runtime/index.js` passes `resetCachedDatabase` into
  `createImportRuntime`.

The restore path (`restoreProductBackup`) now follows the same cached-connection
boundary: it calls `resetCachedDatabase()` and removes destination `-wal`/`-shm`
sidecars before and after copying the verified backup into place.

**Verification**
- Isolated repro: `before` retained ✓, `afterC` removed ✓.
- `backup-restore-smoke`: 3 consecutive PASS.

## #2 — Connection test always reported InnerLife as "paused"

**Severity:** low (misleading status)

**Symptom**
`claracore_connection_test` always returned `modules.innerlife = "paused"`
regardless of the real daemon state.

**Root cause** (`core/gateway/mcp-server.js`)
It read `configuration?.innerlife?.daemon?.status`, but
`database.getConfiguration()` returns an `innerlife` object with **no** `daemon`
key, so the optional chain was always `undefined` and fell back to `"paused"`.

**Fix**
Read the real state via `database.ensureInnerLifeDaemonState(agentId)` and use
its `.status`.

**Verification**
`phase4-gateway-contract-smoke` exercises `connection_test` before its
(pre-existing, unrelated) failure point, so the call now runs without error.

## #3 — Agent identity merge left `current_positions` half-migrated

**Severity:** medium (data inconsistency on rollback)

**Symptom**
`mergeAgentIdentity` migrated ~15 tables inside one transaction, but the
`current_positions.metadata_json` agentId migration ran in JS **after** the
`COMMIT`. If the transaction rolled back, position metadata was already
changed — inconsistent state.

**Fix** (`core/db/database.js`)
- Moved the migration into the `BEGIN … COMMIT` block as a pure-SQL statement:
  `UPDATE current_positions SET metadata_json = json_set(metadata_json,
  '$.agentId', <target>) WHERE json_extract(metadata_json, '$.agentId') =
  <source>;` — now atomic with the rest of the merge.
- The reported `currentPositionMetadataUpdated` count is taken from a
  `COUNT(*)` pre-query.
- Removed the now-unused `updateCurrentPositionAgentMetadata` method and its
  post-commit call.

**Verification**
Repro merging `hermes:lara` → `lara`: metadata became
`{"agentId":"lara","k":1}` (agentId migrated, other keys preserved), count
correct, line `agent_id` migrated.

## #5 - Packaged Gateway / SQLite lock failures

**Severity:** high (frequent agent operation failure)

**Symptom**
Gateway traces intermittently showed:

- `Parse error near line 2: database is locked (5)`
- calls failed in tens of milliseconds, for example `innerlife_status`,
  `innerlife_session_start`, and `shared_line_update`.

**Root cause**
Packaged Electron can fall back to the `sqlite3` CLI when `node:sqlite` is not
available. The built-in SQLite path set WAL and `busy_timeout`, but the CLI
fallback did not. Cross-process contention between Desktop and multiple stdio
Gateway processes therefore failed immediately instead of waiting for the
writer lock to clear.

**Fix** (`core/db/database.js`)
`runSqliteCli()` now launches sqlite3 with:

```
-cmd ".timeout 30000"
```

This gives the fallback path the same contention behavior as the built-in
SQLite connection.

**Verification**
- Later `shared_line_update` calls completed after contention instead of
  failing immediately.
- All-tools temporary MCP pass: 78/78 tools called successfully.

## #6 - Packaged Gateway process blocked replacing `.app`

**Severity:** medium (development packaging friction)

**Symptom**
After quitting Desktop, copying a new build into `/Applications` could still
fail because a packaged Gateway process kept the app bundle executable open:

```
/Applications/ClaraCore Desktop.app/Contents/MacOS/ClaraCore Desktop --gateway
```

**Root cause**
Agent-launched stdio Gateway processes are sibling processes, not children of
the visible Desktop UI. Quitting the UI did not stop those packaged `--gateway`
processes.

**Fix**
- `core/gateway/mcp-server.js` closes its cached database connection and exits
  when stdio transport closes.
- `electron/main.js` best-effort stops sibling packaged Gateway processes on
  UI quit.

**Verification**
- Old packaged Gateway processes were killed before replacing the app.
- After reopening, only the main Desktop process was present.

## #8 - Shared Line current position unique constraint

**Severity:** high (agent Shared Line updates failed)

**Symptom**
Gateway traces showed:

```
Runtime error near line 2: UNIQUE constraint failed: current_positions.line_id (19)
```

This happened when agents updated imported Shared Lines whose existing
`current_positions.id` was a legacy id such as
`old_continuity_position_thread_*`.

**Root cause**
Desktop now enforces one current position per `line_id`, but
`saveCurrentPosition()` inserted `position_${lineId}` and only handled
conflicts on `id`. For imported rows, the existing id did not match
`position_${lineId}`, so SQLite correctly rejected the second current position
for the same line.

**Fix** (`core/db/repositories/continuity.js`)
- Reuse the existing current position id when one exists.
- Upsert `current_positions` by `line_id`.
- Keep history and snapshots append-only.

**Verification**
- `node core/tests/phase3-gateway-smoke.js`: PASS.
- All-tools temporary MCP pass: 78/78 tools called successfully.
- Formal-data-copy test: copied the live product data to a temp data root and
  updated the active Iris line through MCP without triggering the unique
  constraint.

## #9 - `shared_line_update` returned the wrong line

**Severity:** medium (misleading agent state)

**Symptom**
`shared_line_update` could write the active line but return a resume packet for
another line, especially when timestamps were equal or agent-inferred recent
line selection chose `line_default`.

**Root cause**
The Gateway called `saveCurrentPosition(args)` and then re-read
`getResumePacket({ lineId: args.lineId, agentId: args.agentId })`. If
`args.lineId` was omitted, the second read could infer a different line than
the one just written.

**Fix**
- `saveCurrentPosition()` resolves the write target as: explicit `lineId`, else
  current active line.
- `shared_line_update` reads the resume packet using the `lineId` returned by
  the write.
- Ensuring `line_default` exists no longer refreshes its `updated_at` when it is
  already active, so reads do not make it appear artificially recent.

**Verification**
- `node core/tests/phase3-gateway-smoke.js`: PASS.
- All-tools temporary MCP pass: 78/78 tools called successfully.
